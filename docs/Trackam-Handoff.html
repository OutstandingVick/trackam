1 · Executive summary

Trackam is an automatic coding-time tracker — like WakaTime — that is free to self-host and 100% compatible with the official WakaTime editor plugins. It targets developers for whom WakaTime's US$9–14/month subscription is unaffordable, starting with the Nigerian / African developer community.

Because the WakaTime editor plugins are open source and speak a documented HTTP API, Trackam does not ship its own editor extensions. Instead it re-implements the WakaTime ingestion API. Developers keep the plugin they already use (VS Code, Cursor, JetBrains, Vim, etc.) and simply point its api_url at their Trackam server.

What is built today (MVP)

  • WakaTime-compatible heartbeat ingestion (bulk & single) and editor status-bar endpoint.
  • Email/password accounts with per-user API keys.
  • Dashboard: Today / 7-day / 30-day totals, daily chart, and breakdowns by project, language, editor and OS.
  • A distinctive neo-brutalist UI in the brand palette.
  • One-command Docker Compose deployment (app + Postgres + migrations).

2 · Problem & market

WakaTime is the de-facto coding-time tracker, but its pricing puts it out of reach for many developers in emerging markets. The paid tiers are $9/mo (Basic) and $14/mo (Premium); crucially, the free tier limits dashboard history to roughly the last week, so a developer's own historical data is effectively paywalled.

Why the cost feels high

  • The heavy lifting (editor plugins + wakatime-cli) is open source and runs on the user's machine — the subscription mostly pays for hosted dashboards and history retention.
  • Access to your own past data is the primary thing being sold.
  • Priced in USD, the amount is disproportionately large in NGN terms.

Trackam's wedge

Give developers unlimited history and the same automatic tracking for free by self-hosting, with an optional low-cost hosted tier priced in naira later. The addressable audience includes bootcamps, student developer communities, and indie devs across Africa.

3 · How it works (the core idea)

WakaTime editor plugins emit heartbeats — small events describing what you are editing right now (file, project, language, branch, whether you're typing, etc.). The plugin batches these and POSTs them to whatever api_url is configured. Trackam implements the same endpoints those plugins expect, so it is a drop-in server. 

Editor (WakaTime plugin)
   │  batches heartbeats
   ▼
POST /api/v1/users/current/heartbeats.bulk   ── Authorization: Basic base64(api_key)
   │
   ▼
Trackam API (Express)  ──►  PostgreSQL (heartbeats table)
   │
   ▼
Dashboard  ──►  GET /api/stats?range=...  ──►  duration algorithm  ──►  charts
The only thing a developer changes is two lines in ~/.wakatime.cfg: api_url and api_key. No new software to install.

4 · Architecture overview

Trackam is a single Node.js service that serves three concerns from one process:

  • WakaTime-compatible ingestion API (mounted at /api/v1) — authenticated by API key.
  • Account & stats API (/api/account, /api/stats) — authenticated by a signed session cookie.
  • Static frontend — the dashboard SPA served from public/.

State lives entirely in PostgreSQL. There is no background worker or queue; coding-time durations are computed on-read from raw heartbeats, which keeps ingestion cheap and the system easy to reason about at MVP scale.

RuntimeSingle stateless Node process (horizontally scalable behind a load balancer).
StoragePostgreSQL — raw heartbeats + users.
Compute modelDurations derived at query time (no pre-aggregation yet).
Deploy unitDocker image; Compose bundles app + DB + migration job.

8 · The coding-time algorithm

Coding time is derived from raw heartbeats the way WakaTime does it. Heartbeats are sorted by time; the gap between each consecutive pair is added to the total only if it is below a configurable timeout (HEARTBEAT_TIMEOUT_SECONDS, default 900s / 15 min). Larger gaps are treated as a break between sessions, so idle time is not counted. Each interval is attributed to the project / language / editor / OS of the heartbeat that starts it. 

for i in 1..n:
    delta = t[i] - t[i-1]
    if 0 < delta < timeout:
        total += delta
        attribute delta to project/language/editor/os of heartbeat[i-1]

The implementation lives in src/utils/duration.ts. Per-day series are produced by bucketing heartbeats into local-day groups and running the same routine per day. Editor and OS are inferred from the User-Agent string when the plugin doesn't send them explicitly (src/utils/useragent.ts).

This on-read approach is simple and accurate for MVP volumes. At larger scale, add a nightly pre-aggregation (daily summaries table) to keep dashboard queries fast.

9 · Authentication & security

Two auth paths

  • Dashboard: bcrypt-hashed passwords; on login a JWT is signed with JWT_SECRET and stored in an httpOnly cookie (trackam_session, 30-day expiry, secure in production, SameSite=Lax).
  • Ingestion: per-user UUID API key sent via HTTP Basic, validated for UUID shape before a DB lookup.

Current safeguards

  • Parameterized SQL everywhere (no string interpolation).
  • HTML escaping in the dashboard render path.
  • Input validation on signup (email/username/password rules).
  • Unique constraints surface friendly 409s instead of leaking DB errors.
Hardening TODO before scale / public hosting: set a strong JWT_SECRET (the app warns if the insecure default is used); add rate limiting on auth + ingestion; consider hashing API keys at rest; add CSRF protection if any cookie-authed state-changing endpoints are exposed cross-site; put the service behind TLS (reverse proxy).

10 · Frontend & design system

Then sign up in the dashboard and copy the generated ~/.wakatime.cfg snippet into your editor's WakaTime config.

13 · Deployment

Designed for a single cheap VPS (≈1 vCPU / 1 GB RAM). Docker Compose starts three services: db (Postgres), a one-shot migrate job, and the app.

cp .env.example .env          # set a strong JWT_SECRET + BASE_URL
docker compose up -d --build
# dashboard on http://your-host:3000

Recommended production additions

  • Reverse proxy (Caddy/Nginx) for TLS + gzip; Caddy gives automatic HTTPS.
  • Automated Postgres backups (e.g. nightly pg_dump to object storage).
  • Uptime monitoring against /healthz.

16 · Known limitations

  • Durations are computed on every request; no pre-aggregation yet (fine at MVP scale).
  • Tailwind via CDN, not a compiled build.
  • No rate limiting on auth or ingestion endpoints.
  • API keys stored in plaintext (UUIDs) rather than hashed.
  • No email verification / password reset flow.
  • Day bucketing uses server local time, not per-user timezone.
  • No teams, leaderboards, badges, or payments yet (see roadmap).

17 · Roadmap & monetization

Near-term features

  • Private leaderboards — high value for Nigerian dev communities & bootcamps.
  • GitHub README time badges — free organic marketing.
  • Weekly email summaries.
  • Per-user timezones and goals.

Monetization (deferred)

Core stays free/open-source. An optional hosted tier priced in naira via Paystack (cards, bank transfer, USSD) — e.g. ₦1,000–2,000/month — can fund free hosting for those who self-host-averse, without paywalling history.

Scale hardening

  • Daily summary aggregation table + background job.
  • Rate limiting, API-key hashing, CI integration tests.
  • Compiled Tailwind build; optional multi-region.

18 · Onboarding checklist for a new developer

  1. Clone github.com/OutstandingVick/trackam.
  2. npm install; copy .env.example → .env and set JWT_SECRET + DATABASE_URL.
  3. Start Postgres (local or docker compose up db), then npm run migrate.
  4. npm run dev; open the dashboard and sign up.
  5. Point a real editor's WakaTime plugin at http://localhost:3000/api/v1 using your API key, code for a minute, and confirm heartbeats appear.
  6. Read src/routes/wakatime.ts (ingestion) and src/utils/duration.ts (the algorithm) first — they are the heart of the system.
  7. Run npm test before pushing.
